STM32F103 模拟示波器/逻辑分析仪改动记录
========================================

一、项目目标变化
----------------

最初目标是做一个基于 STM32F103 的简易模拟示波器：

1. ADC 采集模拟电压。
2. DMA 保存 ADC 数据。
3. OLED 显示波形。

后续需求扩展为：

1. 能检测 I2C、SPI、UART、PWM 等数字信号。
2. 能在 OLED 上显示多通道逻辑波形。
3. 能暂停、抓帧、左右查看采样数据。
4. 不做协议解码，只判断通信信号是否存在、是否大致正常。


二、最初 ADC/模拟示波器部分
---------------------------

已有代码：

1. AD.c / AD.h
2. Timer.c / Timer.h
3. main.c

检查结果：

1. ADC1 使用 DMA1_Channel1，方向正确。
2. ADC 由 TIM3 TRGO 触发，适合稳定采样。
3. ADC 输入为 PA1 / ADC_Channel_1。
4. TIM3 当前配置可产生固定采样节拍。

主要修改：

1. 将 ADC_Buffer 声明为 volatile，避免 DMA 更新的数据被编译器优化误判。
2. AD.h 中导出 ADC_BUFFER_SIZE 和 ADC_Buffer。
3. main.c 中接入 OLED 显示和波形绘制。

-----------------

原来的 OLED.c 只写了一部分软件 I2C 函数，OLED.h 中声明了很多函数，但 OLED.c 中没有完整实现。

已补全：

1. SSD1306 初始化。
2. 软件 I2C 起始、停止、发送字节。
3. 写命令、写数据。
4. OLED 显存 OLED_GRAM。
5. 清屏 OLED_Clear。
6. 刷屏 OLED_Update。
7. 画点 OLED_DrawPoint。
8. 画线 OLED_DrawLine。
9. 字符显示 OLED_ShowChar。
10. 字符串显示 OLED_ShowString。
11. 数字显示 OLED_ShowNum。
12. 模拟波形显示 OLED_ShowWaveform。
13. 数字逻辑波形显示 OLED_ShowLogicWaveform。

后续优化：

1. 将逻辑波形绘制改成阶梯波显示，即水平线加垂直跳变，不再用斜线连接高低电平。


四、加入逻辑分析仪模式
----------------------

新增文件：

1. Logic.c
2. Logic.h

初版功能：

1. 使用 PA0~PA3 作为 4 路逻辑输入。
2. CH0 = PA0
3. CH1 = PA1
4. CH2 = PA2
5. CH3 = PA3
6. 使用 TIM2 中断定时采样。
7. 每个采样点保存 GPIOA->IDR 的低 4 位。
8. OLED 显示 4 路数字波形。

初始采样方式：

1. 先做自由采样。
2. 后续加入边沿触发。
3. 再加入预触发采样。


五、浮空输入问题
----------------

现象：

没有外接任何信号时，CH0~CH3 波形一直上下跳动。

原因：

PA0~PA3 最初配置为浮空输入，未接线时电平不确定，会被环境噪声影响。

修改：

将 PA0~PA3 改为内部上拉输入：

GPIO_Mode_IPU

结果：

未接信号时通道稳定显示高电平。


六、I2C 测试过程
----------------

测试对象：

另一个 STM32 使用软件 I2C 与 MPU6050 通信。

主控软件 I2C 引脚：

1. PB10 = SCL
2. PB11 = SDA

逻辑分析仪接线：

1. PB10 / SCL -> PA0 / CH0
2. PB11 / SDA -> PA1 / CH1
3. 两块 STM32 必须 GND 共地

MPU6050 接线：

1. STM32 主机 3.3V -> MPU6050 VCC
2. STM32 主机 GND -> MPU6050 GND
3. STM32 主机 PB10 -> MPU6050 SCL
4. STM32 主机 PB11 -> MPU6050 SDA
5. 分析仪 PA0 并联到 SCL
6. 分析仪 PA1 并联到 SDA

观察结果：

1. CH0 出现连续规律脉冲，符合 SCL。
2. CH1 出现不规则变化，符合 SDA。
3. CH2/CH3 未接信号时为直线，正常。
4. 主控 OLED 显示 ID:68，说明 MPU6050 地址读取正常。

结论：

I2C 通信正在发生，SCL/SDA 波形存在，MPU6050 有响应。


七、不做协议解码的判断方式
--------------------------

最终决定：

暂时不做 I2C/SPI/UART 协议解码。

原因：

1. OLED 屏幕太小，不适合显示复杂解码结果。
2. 当前目标是判断通信是否存在、波形是否大致正常。
3. 协议解码需要更高采样率、更稳定触发、更复杂显示。

判断 I2C 是否正常的依据：

1. 空闲时 SCL/SDA 都应为高。
2. 通信时 SCL 应有规律时钟。
3. 通信时 SDA 应有数据变化。
4. 接上 MPU6050 后，主控能读到 ID:68，说明总线通信有效。


八、采样率和显示窗口优化
------------------------

遇到的问题：

波形太密集，在 OLED 上看不出是什么波形。

原因：

采样率、采样点数、OLED 宽度之间存在矛盾。

例如：

1. 采样率 500kHz。
2. 采样点 512。
3. 总采样窗口为 512 / 500000 = 1.024ms。
4. OLED 宽度只有 128 像素。
5. 如果直接显示完整 512 点，波形会挤在一起。

修改：

加入 LOGIC_VIEW_SAMPLE_SIZE：

#define LOGIC_VIEW_SAMPLE_SIZE 64

含义：

1. 实际采集可以是 512 点。
2. OLED 每次只显示其中 64 点。
3. 这样相当于横向放大波形。

可选值：

1. 32：放大更多，适合看边沿。
2. 64：推荐值，细节和范围折中。
3. 128：看更多周期。
4. 256：看更长过程。
5. 512：显示完整采样窗口，但最密。


九、采样率宏优化
----------------

问题：

原先 TIM2 的采样率需要手动修改 Prescaler 和 Period，不方便。

新增接口：

void Logic_SetSampleRate(uint32_t sampleRate);

现在只需要修改 Logic.h 中：

#define LOGIC_SAMPLE_RATE_HZ 500000UL

常用设置：

1. 100000UL：100kHz
2. 200000UL：200kHz
3. 500000UL：500kHz
4. 1000000UL：1MHz

说明：

TIM2 先分频到 1MHz，再用 ARR 产生目标采样率。


十、暂停和抓帧优化
------------------

最初暂停方式：

按键按下后暂停刷新 OLED，保持当前画面。

问题：

暂停时不知道当前画面位于一次通信的哪个位置。

改进：

1. 加入触发。
2. 加入预触发。
3. 加入自动定格。

当前行为：

1. 等待触发边沿。
2. 保留触发前几个采样点。
3. 继续采完整一帧。
4. 采完自动定格。
5. OLED 显示当前帧。
6. 按键后再采下一帧。


十一、从 I2C 专用触发改成通用触发
--------------------------------

曾经做过：

I2C START 触发，即：

SCL 为高时，SDA 从高变低。

问题：

这个模式只适合 I2C，不适合 SPI/UART/PWM。

最终改成：

通用预触发边沿采集。

当前配置：

#define LOGIC_TRIGGER_MODE        LOGIC_TRIGGER_BOTH
#define LOGIC_TRIGGER_CHANNEL     0
#define LOGIC_PRE_TRIGGER_SAMPLES 16

含义：

1. 使用 CH0 作为触发通道。
2. 任意边沿触发。
3. 触发前保留 16 个采样点。
4. 触发后继续采样。

适用场景：

1. I2C：CH0 接 SCL 或 SDA。
2. SPI：CH0 接 SCK，或者改触发通道为 CS。
3. UART：CH0 接 TX，建议用下降沿触发。
4. PWM：CH0 接 PWM，用上升沿或任意边沿触发。


十二、按键浏览完整帧
--------------------

问题：

OLED 屏幕太小，一屏看不完整个通信过程。

解决：

采集完整 512 点，但 OLED 每次只显示 64 点，用按键左右移动显示窗口。

按键定义：

1. PB1：画面往左。
2. PB0：画面往右。
3. PB12：采集下一帧。

接线方式：

PBx -> 按键 -> GND

程序中使用内部上拉。

当前移动步长：

#define LOGIC_VIEW_STEP 8

也就是每按一次，显示窗口移动 8 个采样点。

OLED 顶部显示：

O000、O008、O016 ...

表示当前显示窗口的采样偏移。

十四、不同信号的建议接线和触发
------------------------------

I2C：

1. SCL -> CH0 / PA0
2. SDA -> CH1 / PA1
3. 触发：CH0 任意边沿，或 CH1 下降沿

UART：

1. TX -> CH0 / PA0
2. RX -> CH1 / PA1，可选
3. 建议触发：CH0 下降沿

SPI：

1. SCK -> CH0 / PA0
2. MOSI -> CH1 / PA1
3. MISO -> CH2 / PA2
4. CS -> CH3 / PA3
5. 建议触发：CS 下降沿，即触发通道改为 CH3，触发模式改为 FALLING

PWM：

1. PWM -> CH0 / PA0
2. 建议触发：CH0 上升沿或任意边沿


十五、当前限制
--------------

1. 旧版本曾使用 TIM2 中断采样；当前版本已改为 TIM2 + DMA 采样。
2. DMA 版本降低了 CPU 负担，但采样率、RAM 容量和 OLED 显示宽度仍然有限。
3. OLED 屏幕分辨率有限，只适合低速信号的便携观察。
4. 当前不做协议解码。
5. 只能大致判断通信是否存在、波形是否合理。
6. 对高速 SPI、高速 UART、400kHz I2C 的细节判断能力有限。

二十二、PWM 观察问题说明
----------------------

问题：
测试 PWM_SetCompare1(0)、PWM_SetCompare1(50)、PWM_SetCompare1(100) 时，以为 50% 占空比应该显示成中间电压。

说明：
PWM 的 50% 占空比不是 1.65V 直流电压，而是 0V 和 3.3V 各占一半时间。

实际现象：
1. 0% 占空比：MAX=0，VPP=0。
2. 50% 占空比：MAX=3300，VPP=3300。
3. 100% 占空比：MAX=3300，VPP=0。

结论：
该现象正确，说明 ADC 能采到 PWM 的高低电平。若想得到接近 1.65V 的模拟电压，需要给 PWM 加 RC 滤波。

附录：工程内简化记录
====================

STM32F103 模拟示波器/逻辑分析仪问题与优化记录
================================================

1. OLED 波形闪烁
   问题：连续清屏和刷新导致逻辑分析仪波形闪烁明显。
   优化：逻辑分析仪改为“采集一帧后定格显示”，只有翻页或重新采集时才刷新画面。

2. 未接信号时通道乱跳
   问题：逻辑输入悬空时，PA0~PA3 会受到干扰，OLED 上出现上下摆动。
   优化：逻辑输入配置为上拉输入，未接线时默认保持高电平。

3. I2C 波形太密集，不容易判断通信是否正常
   问题：OLED 宽度有限，连续采样时看不到通信开始位置。
   优化：增加通用边沿触发，检测到 CH0 边沿后开始保存一帧，并保留触发前的若干采样点。

4. 采集完不知道从哪里开始看
   问题：暂停当前画面时，无法确定这一帧处于通信过程中的哪个位置。
   优化：采集后自动定格，并在波形区域显示触发位置标记。

5. 一屏显示不完整
   问题：OLED 一次只能显示有限采样点，完整通信过程可能超出屏幕宽度。
   优化：增加左右查看功能，PB1 向左移动显示窗口，PB0 向右移动显示窗口。

6. 需要手动采集下一帧
   问题：定格后如果想重新观察下一次通信，需要重新开始采集。
   优化：增加 PB11 按键，按下后解除定格，等待下一次边沿并采集新一帧。

7. 不想通过改代码切换模式
   问题：示波器模式和逻辑分析仪模式原本需要通过软件配置切换。
   优化：增加 PA6 模式切换按键，按下后在 SCOPE 和 LOGIC 模式之间切换。

8. main.c 逻辑过多
   问题：示波器显示、逻辑分析仪采集、按键处理、模式切换都堆在 main.c 中，后续维护不方便。
   优化：新建 app/App.c 和 app/App.h，将应用层功能移入 App.c，main.c 只保留 App_Init() 和 App_Process()。

9. PWM 测试理解问题
   问题：误以为 50% 占空比应该显示为 1.65V 或屏幕中间的线。
   说明：PWM 仍然只有 0V 和 3.3V 两种电平，50% 占空比表示高低电平时间各占一半。
   现象：0% 时 MAX=0、VPP=0；50% 时 MAX=3300、VPP=3300；100% 时 MAX=3300、VPP=0，属于正常结果。

二十四、逻辑分析仪采样方式改为 TIM2 + DMA
--------------------------------------

问题：
原来的逻辑分析仪使用 TIM2 更新中断采样，每个采样点都要进入 TIM2_IRQHandler。采样率升高后，中断频率过高，CPU 负担很重，不适合继续提高采样率，也不利于后续做上位机传输。

修改：
1. 逻辑分析仪采样改为 TIM2 更新事件触发 DMA。
2. 使用 DMA1_Channel2，对应 TIM2_UP DMA 请求。
3. DMA 的源地址设置为 GPIOA->IDR。
4. DMA 的目标地址设置为逻辑采样缓冲区。
5. 每次 TIM2 更新事件发生时，DMA 自动读取一次 PA0~PA3 的电平状态。
6. 取消 TIM2_IRQHandler 中逐点采样的方式。

当前采样流程：
1. TIM2 按设定采样率产生更新事件。
2. TIM2 更新事件触发 DMA1_Channel2。
3. DMA 自动把 GPIOA->IDR 搬到 Logic_WorkBuffer 或 Logic_Buffer。
4. DMA 采满一块数据后，主循环检测 DMA 完成标志。
5. 如果是边沿触发模式，CPU 在采样块中查找触发边沿。
6. 找到边沿后，从 DMA 工作缓冲区中截取触发点前后的一帧数据给 OLED 显示。
7. 如果当前采样块没有找到边沿，则自动重新采一块，继续等待触发。

优点：
1. 采样期间不再每个点进入中断。
2. CPU 负担明显降低。
3. 后续更适合把采样数据通过串口或 USB 发给 PC 上位机显示。
4. OLED 可以只作为辅助显示，不再承担完整高速波形查看任务。

当前实现限制：
1. 仍然是定时采样，不是协议解码。
2. GPIO 变化本身不能直接触发 DMA，仍然需要 TIM2 提供固定采样节拍。
3. 触发边沿是在 DMA 采完一块数据后由 CPU 扫描得到，不是硬件实时停止触发。
4. 对短时间偶发信号，采样块长度和采样率仍然需要合理设置。

已验证：
Keil uVision5 / ARMCC 5 编译通过，0 Error(s), 0 Warning(s)。

二十五、触发条件从任意边沿改为空闲后边沿
--------------------------------------

问题：
DMA 采样后在缓冲区中查找任意边沿，虽然能抓到跳变，但这个跳变不一定是通信协议的起始位置，可能只是 I2C/SPI/UART 通信过程中的某个普通数据边沿。

修改：
1. 增加 LOGIC_IDLE_BEFORE_TRIGGER_SAMPLES 宏。
2. 默认要求触发通道在触发前连续稳定 32 个采样点。
3. 只有“稳定空闲之后出现的第一个匹配边沿”才会被认为是触发点。
4. 这样更容易抓到一次通信开始，而不是通信中间的数据跳变。

当前配置：
LOGIC_IDLE_BEFORE_TRIGGER_SAMPLES = 32
LOGIC_TRIGGER_CHANNEL = 0
LOGIC_TRIGGER_MODE = LOGIC_TRIGGER_BOTH

使用说明：
1. I2C：CH0 可接 SCL 或 SDA，建议先接 SCL 观察整段时钟；若要更接近 START，可把 CH0 接 SDA。
2. SPI：如果有 CS，建议后续把触发通道改成 CS，并使用下降沿触发。
3. UART：CH0 接 TX，使用下降沿更接近起始位。
4. 连续 PWM：因为没有明显空闲段，可能无法触发；此时把 LOGIC_IDLE_BEFORE_TRIGGER_SAMPLES 改成 0，恢复普通边沿触发。

已验证：
Keil uVision5 / ARMCC 5 编译通过，0 Error(s), 0 Warning(s)。

二十六、修复边沿触发的 DMA 分块边界问题
------------------------------------

问题：
边沿触发在 DMA 采完一块数据后再扫描触发点。这样有两个边界问题：
1. 如果真正的通信起始边沿落在 DMA 块后半段，当前块后面可能不够一整帧数据，原逻辑会丢掉这个边沿。
2. 如果通信起始刚好跨两个 DMA 块，下一块看不到上一块末尾的空闲电平，可能无法满足“空闲后触发”的判断。

修改：
1. 增加 Logic_HistoryBuffer 历史缓冲区。
2. 每次没有找到有效触发时，保留上一段采样末尾 512 点。
3. 下一次扫描时，把“历史缓冲区 + 当前 DMA 缓冲区”当作连续采样数据一起判断。
4. 触发边沿、触发前空闲、触发前预采样、触发后数据都可以跨 DMA 块边界。

效果：
1. 降低真实通信起始边沿被 DMA 分块边界丢掉的概率。
2. “空闲后第一个边沿”判断更稳定。
3. 后续上位机显示时，也可以沿用这种连续采样思路。

已验证：
Keil uVision5 / ARMCC 5 编译通过，0 Error(s), 0 Warning(s)。

二十七、重新划分五个按键功能
--------------------------

需求：
使用 5 个独立按键控制示波器/逻辑分析仪和协议触发模式。

新按键分配：
1. PC14：切换示波器模式和逻辑分析仪模式。
2. PA4：切换逻辑分析仪触发协议模式，循环为 IIC、SPI、UART。
3. PB10：采集下一帧。
4. PB0：逻辑分析仪显示窗口左移。

OLED 显示：
逻辑分析仪顶部会显示当前触发协议模式：IIC、SPI 或 UART，同时显示采样率、显示窗口宽度和触发通道。

当前默认触发配置：
1. IIC：CH1 触发，下降沿，建议 CH0 接 SCL，CH1 接 SDA。
2. SPI：CH3 触发，下降沿，建议 CH3 接 CS。

二十八、IIC 模式改为严格 START 条件触发
-----------------------------------

问题：
原来的 IIC 模式本质是 CH1/SDA 下降沿触发。虽然很多情况下能抓到 IIC 起始，但它没有检查 SCL 是否为高电平，因此只能算“像 START”，不能严格判断为 IIC START。

说明：
IIC 起始条件是：SCL 保持高电平时，SDA 从高电平变为低电平。因此看到 SDA 先于 SCL 时钟脉冲跳变，本身不是错误，反而是 START 条件的正常特征。

修改：
1. 新增 LOGIC_TRIGGER_IIC_START 触发类型。
2. IIC 模式不再只判断 SDA 下降沿。
3. 现在要求 CH0/SCL 在 SDA 下降前后都为高电平。
4. 现在要求触发前一段采样中 CH0/SCL 和 CH1/SDA 都保持高电平空闲。
5. 满足以上条件后，才认为该边沿是 IIC START。

默认接线：
CH0 / PA0 -> SCL
CH1 / PA1 -> SDA
GND -> GND

已验证：
Keil uVision5 / ARMCC 5 编译通过，0 Error(s), 0 Warning(s)。

二十九、当前版本阶段性改动整理
----------------------------

当前版本定位：
本项目已经从最初的简易模拟示波器，扩展为“模拟示波器 + DMA 逻辑分析仪”的双模式工具。OLED 主要用于现场辅助观察，后续可以继续加入 PC 上位机显示完整采样数据。

一、工作模式
1. 示波器模式：使用 PA1 / ADC1_CH1 输入，通过 ADC + DMA 采样并在 OLED 上显示波形、MAX 和 VPP。
2. 逻辑分析仪模式：使用 PA0~PA3 作为 CH0~CH3，通过 TIM2 + DMA1_Channel2 定时读取 GPIOA->IDR。
3. PC14 按键用于切换示波器模式和逻辑分析仪模式。

二、逻辑分析仪采样方式
1. 旧版本使用 TIM2 中断采样，每个采样点都进入中断，采样率升高后 CPU 负担较大。
2. 当前版本改为 TIM2 更新事件触发 DMA。
3. DMA 源地址为 GPIOA->IDR，目标地址为逻辑采样缓冲区。
4. 采样过程中 CPU 不再逐点进入中断，只在 DMA 采完一块后处理数据。
5. 这为后续串口或 USB 上位机传输打下基础。

三、触发方式
1. 当前逻辑分析仪不是协议解码，而是协议相关触发。
2. PA4 按键用于切换 IIC、SPI、UART 三种触发模式。
3. IIC 模式：默认 CH0=SCL，CH1=SDA，使用严格 START 条件触发。
4. SPI 模式：默认 CH3=CS，使用下降沿触发。
5. UART 模式：默认 CH0=TX，使用下降沿触发。

四、IIC START 触发修正
1. 之前 IIC 模式只是检测 SDA 下降沿，不能保证是真正 START。
2. 当前版本新增 LOGIC_TRIGGER_IIC_START。
3. 判断条件为：SCL 为高时，SDA 从高变低。
4. 同时要求触发前一段采样中 SCL 和 SDA 都保持高电平空闲。
5. 因此看到 SDA 先于 SCL 时钟脉冲跳变，本身是 IIC START 的正常特征。

五、DMA 分块边界修复
1. DMA 是分块采样，如果通信起始边沿刚好落在块边界附近，可能被旧逻辑漏掉。
2. 当前版本增加 Logic_HistoryBuffer。
3. 每次未触发时保留上一块末尾数据。
4. 下一次扫描时将“历史缓冲区 + 当前 DMA 缓冲区”作为连续数据判断。
5. 这样能降低真实通信起始边沿被 DMA 分块边界丢掉的概率。

六、非阻塞采集修复
1. 旧版本等待触发时使用 while 阻塞等待 Logic_IsCaptureDone()。
2. 如果信号没有满足触发条件，主循环会被卡住，按键无法响应。
3. 当前版本改为非阻塞采集。
4. 等待触发期间仍然持续检测 PC14、PA4、PB10、PB0、PA6。
5. 因此即使暂时没有触发，也可以切换模式或重新采集。

七、当前按键分配
1. PC14：切换示波器/逻辑分析仪模式。
2. PA4：切换 IIC/SPI/UART 触发模式。
3. PB10：采集下一帧。
4. PB0：逻辑分析仪显示窗口左移。
5. PA6：逻辑分析仪显示窗口右移。

八、当前默认接线建议
IIC：
1. CH0 / PA0 -> SCL
2. CH1 / PA1 -> SDA
3. GND -> GND

SPI：
1. CH0 / PA0 -> SCK
2. CH1 / PA1 -> MOSI
3. CH2 / PA2 -> MISO
4. CH3 / PA3 -> CS
5. GND -> GND

UART：
1. CH0 / PA0 -> TX
2. CH1 / PA1 -> RX，可选
3. GND -> GND

九、当前限制
1. OLED 分辨率有限，只适合辅助观察。
2. 当前 IIC/SPI/UART 是触发模式，不是完整协议解码。
3. 逻辑采样率、RAM 容量和 OLED 显示宽度仍然限制可观察的协议速度。
4. 连续 PWM 没有明显空闲段时，可能不适合空闲后触发；可将 LOGIC_IDLE_BEFORE_TRIGGER_SAMPLES 改为 0 恢复普通边沿触发。
5. 后续建议加入 PC 上位机，将 DMA 采样数据发送到电脑端显示和解码。
